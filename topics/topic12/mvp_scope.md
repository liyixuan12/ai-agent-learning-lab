# MVP 范围定义（In-Scope / Out-of-Scope）

## 1. MVP 决策原则

- 只保留“首次价值到达”必需能力（First Value Moment）。
- 每个功能都要对应一个可测指标，不可测则延后。
- 对高风险能力默认延后，先做安全可控版本。

---

## 2. 功能优先级表


| 功能              | 优先级 | 是否进 MVP | 原因          |
| --------------- | --- | ------- | ----------- |
| 双公司对比分析（核心问答）   | P0  | 是       | 直接对应核心场景与价值 |
| 输出附引用与免责声明      | P0  | 是       | 关系可信度与合规边界  |
| 基础异常处理（超时、缺数据）  | P0  | 是       | 保证可用性与稳定体验  |
| 行业模板化提问（如银行/消费） | P1  | 是       | 提升上手效率      |
| 会话摘要与续问建议       | P1  | 否（下一阶段） | 非首次价值必需     |
| 多语言输出           | P2  | 否       | 当前用户群体不强需求  |
| 实时行情与交易信号       | P2  | 否       | 风险高且超出学习目标  |


---

## 3. 明确不做项（Out-of-Scope）

- 个性化买卖建议、收益预测承诺。
- 自动下单或交易接口联动。
- 覆盖所有金融子领域（先聚焦公司基础面分析）。

---

## 4. 版本切分建议

- **v0（本期）**：P0 全部上线，确保“可用 + 可控 + 可讲”。
- **v1（下期）**：加入 P1 的模板化与会话增强。
- **v2（后续）**：在风控与评测成熟后再探索更复杂能力。

---

## 5. 范围冻结规则

- 冻结点后新增需求必须满足：
  - 有明确用户价值证据；
  - 不影响当前验收指标；
  - 有对应的测试与护栏方案。
